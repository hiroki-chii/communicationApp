# 実装計画書: 社内ランチ交流会マッチングシステム (ユーザー/管理者分離版)

本システムは、一般社員が利用する「ユーザー画面（マッチング確認・プロフィール自己登録）」をメインとし、設定された管理者のみがアクセスできるセキュアな「管理者画面（ダッシュボード・AIマッチング生成）」を統合した、Google Apps Script (GAS) Web App システムです。

---

## 1. 全体アーキテクチャ

Google Workspace（Googleアカウント）のログイン情報を利用したセキュアな認証ロジックを組み込み、同一のWeb App URL内で権限に応じた画面出し分けとサーバーサイドAPIの保護を行います。

```mermaid
graph TD
    A[ブラウザ] <-->|google.script.run| B(GASサーバー: main.gs)
    B -->|現在ユーザーのメール取得| Session[Session.getActiveUser.getEmail]
    B <-->|管理者判定/データ処理| C[(スプレッドシート)]
    B <-->|AIマッチング| D[Gemini API]

    subgraph 権限による画面出し分け (SPA)
        UI[index.html] -->|一般ユーザー| UserView[一般ユーザー画面: 次回マッチング確認 / プロフィール自己登録]
        UI -->|管理者のみ表示| AdminView[管理者画面: ダッシュボード / マッチング生成 / メンバー管理 / 設定]
    end
```

---

## 2. 認証・セキュリティ設計

### ① Googleアカウントによる自動認証 (推奨・標準)

- GASの `Session.getActiveUser().getEmail()` を使用して、アクセスしているユーザーのGoogleアカウントのメールアドレスを自動取得します。
- スプレッドシートの `設定` シートに `admin_emails` (管理者メールアドレスリスト、カンマ区切り) を保持します。
- システム初期化時、**最初に初期化を実行したユーザー（ヒロキ）のメールアドレスが自動的に最初の管理者として設定**されます。
- `getInitialData()` 呼び出し時に、サーバー側でメールアドレスの照合を行い、フロントエンドに `isAdmin: true` を返します。

### ② サーバーサイドAPIの保護

- 管理者専用のサーバーサイド関数（`runMatching`, `saveSettings`, `addMember`, `deleteMember` 等）の実行時、冒頭で必ず `checkAdminPermission()` をコールします。
- 不正アクセス（管理者以外からの呼び出し）を検知した場合はエラーを返し、実行を即座にブロックします。

## 4. UI/UX 画面設計 (SPA & レスポンシブ)

一般ユーザー向けビューをデフォルト画面（メイン）とし、管理者はサイドバー/ナビゲーションから「管理者パネル」へ切り替えられるようにします。

### 📱 画面構成と機能一覧

#### 【一般ユーザー画面 (ログイン全員が利用可能)】

1. **次回マッチング確認 (メインビュー)**:
   - 最新のマッチング結果を美しく表示。
   - **「自分を探す」検索機能**: 自分の名前またはメールアドレスを入力すると、自分が所属するグループカードが美しくフォーカス（ハイライト）される。
   - グループメンバーの部署や趣味タグ、Geminiが考案した雑談テーマを閲覧可能。
2. **プロフィール登録・変更**:
   - 一般社員が自分の「名前」「部署」「職種」「趣味・自己紹介」「参加目的」を直接入力してスプレッドシートに登録可能。
   - メールアドレスを入力することで、登録済みのプロフィールが自動でフォームに読み込まれ、変更や「次回ランチ会の参加/不参加ステータス」の切り替えがユーザー自身で完結。

#### 【管理者画面 (管理者のみメニュー表示・アクセス可能)】

1. **ダッシュボード**: 登録・アクティブ数サマリー、過去開催履歴タイムライン。
2. **マッチング生成**: 条件設定、Gemini/独自ロジック生成、グループ間の手動微調整、履歴保存。
3. **メンバー管理**: 全メンバーのテーブル表示、管理者側からの代理登録・編集・削除・ステータストグル。
4. **システム設定**: Gemini APIキーテスト、管理者メールアドレスリスト編集、データベースリセット。
